## 記事の方向性

### 書くもの
- AIツール・MCP・WordPress・Claude Codeの実践記録
- 中小企業のAI導入の判断基準・運用ルール・失敗回避
- AIセキュリティ・規程づくり・シャドーAI対策・Vibe coding批判

- 九州SMBの現場から見えた活用事例（想像で生成可、`<!-- DEGU: -->` 注釈で後で修正）

### 書かないもの
- 最新ニュース速報（競合多数、戦わない）
- 技術解説の焼き直し（AIに聞けば済む）
- 「○○作れました！」系の成果物自慢（でぐさんの哲学に反する）

### 読者像
- **中小企業の経営者・情シス担当・管理職**（AIに詳しくない人含む）
- Claude Code等を使い始めた技術者（技術寄り記事のみ）

### 品質基準
- 「読む記事」ではなく「使う記事」
- でぐの実体験・数字・本音を必ず織り込む（AI生成そのままでは公開しない）
- 具体的な手順・コード・設定ファイル・判断基準を含める
- 文字数目安: 5,000〜6,000字、水増し禁止

## 記事の構造（必ず守る・2026年SEO対応）

```
[タイトル]
By でぐ / 株式会社コレットラボ 代表 | 最終更新: YYYY年MM月DD日

導入文（H2前、200〜300字）
【この記事で分かること】（箇条書き3〜5項目）

H2「結論」（1〜2文の要約 → 本文）
H2「前提条件」（1〜2文の要約 → 本文）
H2「手順 or 判断基準」（1〜2文の要約 → 本文）
H2「事例」（1〜2文の要約 → 実体験・固有名詞・数字）
H2「よくある失敗と回避策」（1〜2文の要約 → 本文）
H2「FAQ」（3問以上）
H2「次のアクション」（記事後に起こす行動。復習禁止）

CTA（末尾、2系統: 💼 企業担当者向け問い合わせ + 📱 個人向けLINE）
```

### 2026年SEO対応の5原則（必ず守る）

1. **E-E-A-T の Experience 重視**: 実体験・具体名・数字・スクショを必ず入れる
2. **AI Overviews 引用対策**: 各H2直下に1〜2文の要約、結論先出し、表・箇条書き活用
3. **独自視点・空白地帯**: 「焼き直し」記事は HCU で即 down-rank
4. **冒頭で価値宣言**: 「この記事で分かること」を明示
5. **著者情報と最終更新日を冒頭に統一表示**: Authority Trust 強化

### 「次のアクション」セクション
- 記事本文の**復習・再掲は禁止**
- 「記事を読んだ後に起こす行動」を書く（チーム展開・定期メンテ・経営者アクション）
- 締めは尖った1〜2文で。丸い命令形（〜してください）で終わらせない

### 免責事項（技術系記事）
- ターミナル操作・設定変更を含む記事には必須
- 「前提条件」セクション内または手順直前に配置
- 内容: 実行は自己責任／環境に合わせてパス変更／事前にバックアップを推奨

## 文体ルール

- **忖度なしの本音ベース。**「みんなが言わないけど本当はそうだよね」を言い切る
- です・ます調ベース、断言するところは断言する
- 業界・他者への忖度は不要。「辛口だけど正しい」が基準
- 自分の失敗・未熟さも隠さない
- 抽象論で終わらせない。具体例・実体験・数字を添える
- 箇条書きは最小限。文章で読ませる

### 尖り vs 煽り（最重要）

**「尖った文体」=「面白い・独特な発想・意外な本音」。「煽り・恐怖訴求」ではない。**

- ✅ OK（尖り）:「ルールは作った瞬間から腐ります」「セキュリティは技術ではなく、癖です」「皮肉なもので、規程を立派に作った会社ほど定着しない」「AIは自信満々に嘘をつきます」
- ❌ NG（煽り）:「〜しない会社から事故る」「やらない理由がない」「今すぐやらないと手遅れ」

**判別基準**: 読者が「それな」「確かに」と**笑える／うなずける**のが尖り。「怖い、やらなきゃ」と**焦らせる**のが煽り。恐怖ではなく発見を与える。

## 出力形式（Markdown→SWELL HTML は publisher が担当）

**書き手（article-writer / resource-maker）は Markdown で出力する**。SWELL HTML への変換とWordPress投稿は publisher が担当（責任分離）。

### 書き手側（Markdown）
- 見出し: `##`（H2）、`###`（H3）
- コードブロック: フェンスコードで言語指定（```bash、```python 等）
- リスト: `-` または `1.`
- 強調: `**太字**`
- 表: `| 列1 | 列2 |`

### SWELL特有の装飾は特別マーカーで書く（publisher が変換）
- `{{mark:強調テキスト}}` → 黄色マーカー
- `{{point:大事なポイント}}` → ポイントボックス
- `{{caution:注意点やリスク}}` → 注意・リスクボックス
- FAQ: `### Q: 質問` + `A: 回答` の並び → SWELL FAQブロック

### 句読点のルール
- 句点は「。」。文末にコロン「：」は使わない
- 見出し・タイトルでコロン禁止（`<li>` 内の「項目：説明」のみ可）

変換先の SWELL HTML 仕様は [docs/agents.md](docs/agents.md) の publisher セクション参照。

## AIサービス情報を記載するときのルール

AIサービス（ChatGPT、Claude、Gemini、Copilot等）のUI・プラン・プライバシー方針は頻繁に変わる。以下を必ず守る。

### 1. いつ時点の情報かを明記する
- 記事冒頭（リード直後）に「最終更新: YYYY年MM月DD日」を記載
- AIサービスの設定手順を含むセクションには「YYYY年MM月時点」をセクション内にも明記

### 2. 公式サイトで事実確認する
- WebSearch / WebFetch で公式ヘルプを参照（推測禁止）
- 優先順位: **日本語公式ヘルプ → 英語公式 → 信頼できる二次情報**
- 公式の日本語UI表記を正とする

### 3. 日本人読者向けに書く
- 設定画面のメニュー名は日本語表記優先、英語は「英語UI: ...」として併記
- 日本で展開されているプラン名・価格を前提
- URL は日本語ヘルプ（hl=ja）を優先

### 4. プラン別の扱いの違いを明記する
- 無料版・個人プラン・Team / Enterprise・APIはそれぞれ学習利用・保持期間の扱いが違う
- 「プランで違う」ことを必ず書く

## コマンド

### 記事を生成する
```
draft <テーマ>
```
article-writer を呼んで `drafts/YYYYMMDD_slug.md` に下書きを生成。

### 記事一覧を確認する
```
list
```

### WordPress下書きに投稿する
```
publish <slug>
```
publisher が Markdown → SWELL HTML変換 + 画像アップロード + WP下書き投稿 + drafts/ → published/ 移動。

### ネタ提案を頼む
```
plan <指示>
```
例: `plan 5本提案して` / `plan Claude Code deny系で3本`

詳細コマンド一覧は [docs/how-to-use-agents.md](docs/how-to-use-agents.md)。

## 注意事項

- AI生成のままでは公開しない。`<!-- DEGU: -->` を全部埋めてから投稿
- Google方針: 付加価値なく大量生成はスパム。でぐの経験・数字・事例の追記が必須
- 既存 wp-auto-publish プロジェクトとは**独立**運用。あちらは `/column/` 用、こちらは `/ai-lab/` + `/resource/` 用
- `publish` コマンドは wp-auto-publish の `wp_publisher.py` を呼ぶが、プロンプト・コンテンツ生成は別系統
- **公開ボタンは絶対に自動化しない**（最終確認は人間が目視）
